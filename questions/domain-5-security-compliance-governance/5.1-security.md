## Q345

### 🇺🇸 Question (English)
Which principle is FUNDAMENTAL when configuring IAM for AI workloads?

### 🇰🇷 문제 (한글)
AI 워크로드에 대해 IAM을 설정할 때 근본적인 원칙은 무엇입니까?

### Options / 보기
- **A.** Grant only the permissions required for a specific job or function.
  - 한글: 특정 작업이나 기능에 필요한 권한만 부여.
- **B.** Grant broad permissions by default and tighten only after issues arise.
  - 한글: 기본으로 넓은 권한을 부여하고 문제가 생긴 뒤에만 좁힘.
- **C.** Rely on network IP allow-lists as the sole authorization mechanism.
  - 한글: 네트워크 IP 허용 목록을 유일한 인가 메커니즘으로 사용.
- **D.** Embed long-lived static access keys directly in application code.
  - 한글: 장기 유효 정적 액세스 키를 애플리케이션 코드에 직접 포함.

### ✅ Correct Answer / 정답
**A**

### 💡 Explanation / 해설

**English**:
Least privilege means IAM roles and policies grant only the permissions required for a specific job or function — nothing more. This limits blast radius when credentials leak, a role is misused, or an application is compromised.

- B is the opposite anti-pattern ("broad first, tighten later"). In practice excess permissions are rarely pruned, and any breach immediately inherits the broad scope.
- C confuses network reachability with authorization. IP allow-lists can be bypassed (spoofing, compromised bastion, on-VPC attacker), and they lack per-principal granularity. They supplement — never replace — IAM.
- D is a severe anti-pattern. Static keys in source code leak through repos, backups, and logs. The correct pattern is IAM roles delivering short-lived STS credentials to the workload.

**한글**:
최소 권한이란 IAM 역할·정책이 특정 작업/기능에 필요한 권한만 부여한다는 원칙입니다. 자격 증명 유출·역할 오용·애플리케이션 침해 시 폭발 반경을 제한합니다.

- B는 반대 안티 패턴("우선 넓게, 나중에 좁힘")입니다. 실제로 초과 권한은 좀처럼 회수되지 않으며, 침해 시 넓은 범위가 그대로 상속됩니다.
- C는 네트워크 도달성을 인가로 혼동합니다. IP 허용 목록은 우회 가능(스푸핑, 침해된 배스천, VPC 내 공격자)하고 주체별 세밀도가 없어 IAM을 대체할 수 없고 보완할 뿐입니다.
- D는 심각한 안티 패턴입니다. 소스에 정적 키를 심으면 저장소·백업·로그를 통해 유출됩니다. 워크로드에는 STS 단기 자격을 전달하는 IAM 역할을 사용해야 합니다.

## Q346

### 🇺🇸 Question (English)
Which statement about SageMaker data encryption is TRUE by default?

### 🇰🇷 문제 (한글)
SageMaker 데이터 암호화의 기본 동작에 관한 진술 중 옳은 것은 무엇입니까?

### Options / 보기
- **A.** Data is transmitted and stored as plain text unless the customer adds encryption.
  - 한글: 고객이 직접 암호화를 추가하지 않는 한 데이터는 평문으로 전송·저장됨.
- **B.** Data is encrypted at rest and in transit using AWS-managed keys.
  - 한글: AWS 관리형 키를 사용해 데이터가 저장 중·전송 중 모두 암호화됨.
- **C.** Only data in transit is encrypted; artifacts written to disk remain in plain text.
  - 한글: 전송 중 데이터만 암호화되고 디스크에 쓰인 아티팩트는 평문으로 남음.
- **D.** Encryption is only available on customer-supplied hardware modules.
  - 한글: 암호화는 고객이 제공한 하드웨어 모듈에서만 사용 가능.

### ✅ Correct Answer / 정답
**B**

### 💡 Explanation / 해설

**English**:
SageMaker uses KMS integration by default: notebook storage, training input/output, model artifacts, and endpoints encrypt data at rest, and API traffic uses TLS in transit. Customers can supply their own KMS Customer Managed Keys (CMKs) for tighter control, but no customer action is required to get baseline encryption.

- A is false because both directions are protected by default.
- C is a partial-truth trap — encryption at rest is also on.
- D confuses KMS with CloudHSM. KMS provides the default; CloudHSM is a separate FIPS 140-2 Level 3 option for organizations needing dedicated HSMs.

**한글**:
SageMaker는 기본으로 KMS 통합을 사용합니다. 노트북 스토리지, 학습 입출력, 모델 아티팩트, 엔드포인트가 저장 시 암호화되고 API 트래픽은 TLS로 보호됩니다. 더 엄격한 통제가 필요하면 자체 KMS Customer Managed Key(CMK)를 제공할 수 있으며, 기본 암호화 자체는 고객 조치가 필요 없습니다.

- A는 두 방향 모두 기본 보호되므로 사실 아닙니다.
- C는 부분 진실 함정 — 저장 시 암호화도 켜져 있습니다.
- D는 KMS와 CloudHSM을 혼동한 설명입니다. KMS가 기본이며, CloudHSM은 전용 HSM이 필요한 조직을 위한 FIPS 140-2 L3 별도 옵션입니다.

## Q347

### 🇺🇸 Question (English)
Which AWS technology lets VPC workloads reach Bedrock or SageMaker over the AWS backbone, never crossing the public internet?

### 🇰🇷 문제 (한글)
VPC 워크로드가 공용 인터넷을 거치지 않고 AWS 백본으로 Bedrock 또는 SageMaker에 도달할 수 있게 하는 AWS 기술은 무엇입니까?

### Options / 보기
- **A.** Amazon CloudFront distributing responses through edge locations.
  - 한글: Amazon CloudFront를 통해 엣지 로케이션에서 응답을 배포.
- **B.** AWS Direct Connect used on its own as the connectivity method.
  - 한글: AWS Direct Connect 자체만을 연결 수단으로 사용.
- **C.** AWS PrivateLink with interface VPC endpoints for the target service.
  - 한글: 대상 서비스에 대한 인터페이스 VPC 엔드포인트가 있는 AWS PrivateLink.
- **D.** Amazon Route 53 private hosted zones resolving service endpoints internally.
  - 한글: 서비스 엔드포인트를 내부에서 이름 해석하는 Amazon Route 53 프라이빗 호스팅 영역.

### ✅ Correct Answer / 정답
**C**

### 💡 Explanation / 해설

**English**:
PrivateLink presents an AWS service as an Elastic Network Interface inside the VPC. Traffic to Bedrock or SageMaker flows over the AWS backbone, never touching the public internet or requiring a NAT gateway.

- A (CloudFront) is a CDN for user-facing content distribution — it doesn't provide private connectivity to backend AWS APIs.
- B (Direct Connect) is a private physical/leased link between on-premises and AWS. It carries traffic privately from the datacenter but by itself doesn't create a private path into a specific service; it typically is combined with PrivateLink to reach service endpoints.
- D (Route 53 private zones) only handles DNS resolution. Resolving a name inside a VPC does not change where the traffic actually goes; the underlying route is still public unless PrivateLink is in place.

**한글**:
PrivateLink는 AWS 서비스를 VPC 내 Elastic Network Interface로 노출합니다. Bedrock·SageMaker 트래픽이 AWS 백본으로만 흘러 공용 인터넷을 거치지 않고 NAT 게이트웨이도 필요 없습니다.

- A(CloudFront)는 사용자용 콘텐츠 배포용 CDN이며 백엔드 AWS API로의 프라이빗 연결을 제공하지 않습니다.
- B(Direct Connect)는 온프레미스와 AWS 사이의 물리 전용선입니다. 데이터센터에서 AWS까지 트래픽을 프라이빗하게 전달하지만 그 자체로 특정 서비스로의 프라이빗 경로를 만들지 않고, 대개 PrivateLink와 조합해 서비스 엔드포인트에 도달합니다.
- D(Route 53 프라이빗 영역)는 DNS 이름 해석만 담당합니다. VPC 내에서 이름을 해석해도 실제 트래픽 경로는 바뀌지 않으며, PrivateLink가 없으면 여전히 공용 경로로 나갑니다.

## Q348

### 🇺🇸 Question (English)
An interface VPC endpoint for Amazon Bedrock is created in a private subnet. Which statement is TRUE about the workloads in that subnet?

### 🇰🇷 문제 (한글)
Amazon Bedrock용 인터페이스 VPC 엔드포인트가 프라이빗 서브넷에 생성되었습니다. 해당 서브넷의 워크로드에 대해 옳은 진술은 무엇입니까?

### Options / 보기
- **A.** Bedrock inference becomes exempt from per-token pricing for the account.
  - 한글: Bedrock 추론이 해당 계정에서 토큰당 요금 대상에서 제외됨.
- **B.** IAM authorization is bypassed for traffic that arrives through the endpoint.
  - 한글: 엔드포인트로 들어온 트래픽에 대해 IAM 인가가 우회됨.
- **C.** Foundation model weights are downloaded into the VPC and served locally.
  - 한글: 파운데이션 모델 가중치가 VPC로 다운로드되어 로컬에서 서빙됨.
- **D.** Workloads reach Bedrock without a NAT gateway or public IP, and IAM still applies.
  - 한글: 워크로드가 NAT 게이트웨이나 퍼블릭 IP 없이 Bedrock에 도달하며 IAM은 계속 적용됨.

### ✅ Correct Answer / 정답
**D**

### 💡 Explanation / 해설

**English**:
An interface VPC endpoint puts Bedrock inside the VPC's network path. Workloads no longer need internet egress (NAT gateway, public IP), yet every request is still authorized by IAM policies attached to the caller, the endpoint policy, and the resource — nothing about VPC access changes the authorization model.

- A confuses network topology with billing. PrivateLink and per-token FM billing are separate concerns; usage still incurs per-token charges.
- B is the classic misconception that "private = trusted". Network isolation and identity-based authorization are independent layers; both must be present.
- C is factually incorrect. Bedrock is a managed inference service; models remain in the service plane. The VPC endpoint only provides a private network path to the same managed inference.

**한글**:
인터페이스 VPC 엔드포인트는 Bedrock을 VPC 네트워크 경로 안에 위치시킵니다. 워크로드는 인터넷 이그레스(NAT 게이트웨이, 퍼블릭 IP)가 필요 없지만, 모든 요청은 여전히 호출자의 IAM 정책, 엔드포인트 정책, 리소스 정책으로 인가됩니다. VPC 접근이 인가 모델을 바꾸지 않습니다.

- A는 네트워크 토폴로지와 청구를 혼동합니다. PrivateLink와 FM의 토큰당 청구는 별개이며, 사용량에는 여전히 토큰당 요금이 부과됩니다.
- B는 "프라이빗 = 신뢰 가능"이라는 전형적 오해입니다. 네트워크 격리와 아이덴티티 기반 인가는 독립적인 계층이며 둘 다 필요합니다.
- C는 사실이 아닙니다. Bedrock은 관리형 추론 서비스이며 모델은 서비스 플레인에 남습니다. VPC 엔드포인트는 동일한 관리형 추론으로의 프라이빗 네트워크 경로만 제공합니다.

## Q349

### 🇺🇸 Question (English)
Which AWS service centrally manages the keys used to encrypt data at rest across SageMaker artifacts, S3, EBS, and other AWS services?

### 🇰🇷 문제 (한글)
SageMaker 아티팩트·S3·EBS 등 AWS 서비스 전반의 저장 중 데이터를 암호화하는 데 사용되는 키를 중앙에서 관리하는 AWS 서비스는 무엇입니까?

### Options / 보기
- **A.** AWS KMS.
  - 한글: AWS KMS.
- **B.** AWS Certificate Manager.
  - 한글: AWS Certificate Manager.
- **C.** AWS Secrets Manager.
  - 한글: AWS Secrets Manager.
- **D.** AWS CloudHSM.
  - 한글: AWS CloudHSM.

### ✅ Correct Answer / 정답
**A**

### 💡 Explanation / 해설

**English**:
AWS KMS is the standard, integrated key management service. Nearly every AWS storage/analytics/ML service — including SageMaker, S3, EBS, RDS, and Bedrock — supports encryption keyed by AWS-owned, AWS-managed, or customer-managed KMS keys with unified auditing through CloudTrail.

- B (Certificate Manager, ACM) manages TLS/SSL certificates for encryption in transit — not the data-at-rest keys used by storage services.
- C (Secrets Manager) stores and rotates secrets like database credentials or API tokens. It is not the encryption key store for AWS services.
- D (CloudHSM) is a single-tenant FIPS 140-2 Level 3 HSM. It can back keys used by KMS via custom key stores, but it is not the standard central service most workloads use — KMS is.

**한글**:
AWS KMS는 표준 통합 키 관리 서비스입니다. SageMaker·S3·EBS·RDS·Bedrock 등 대부분의 AWS 스토리지/분석/ML 서비스가 AWS 소유·AWS 관리·고객 관리 KMS 키로 암호화를 지원하며 CloudTrail로 통합 감사됩니다.

- B(Certificate Manager, ACM)는 전송 중 암호화용 TLS/SSL 인증서를 관리하며, 스토리지 서비스의 저장 시 데이터 키가 아닙니다.
- C(Secrets Manager)는 DB 자격 증명·API 토큰 같은 비밀을 저장·순환합니다. AWS 서비스의 암호화 키 저장소가 아닙니다.
- D(CloudHSM)는 단일 테넌트 FIPS 140-2 L3 HSM입니다. 커스텀 키 스토어로 KMS 키를 뒷받침할 수 있지만 대부분의 워크로드가 사용하는 표준 중앙 서비스가 아닙니다 — KMS가 표준입니다.

## Q350

### 🇺🇸 Question (English)
A team stores training data and model artifacts in S3. Which control PREVENTS accidental public exposure by overriding any bucket or object permission that would otherwise grant public access?

### 🇰🇷 문제 (한글)
학습 데이터와 모델 아티팩트를 S3에 저장한 팀이, 퍼블릭 접근을 허용할 수 있는 버킷·객체 권한을 무효화하여 우발적 노출을 방지하는 통제는 무엇입니까?

### Options / 보기
- **A.** IAM Access Analyzer scheduled to run on the account weekly.
  - 한글: 계정에 대해 매주 실행되도록 예약된 IAM Access Analyzer.
- **B.** S3 Block Public Access at the bucket or account level.
  - 한글: 버킷 또는 계정 수준의 S3 Block Public Access.
- **C.** An S3 bucket policy that grants read access to `Principal: "*"`.
  - 한글: `Principal: "*"`에 읽기 권한을 부여하는 S3 버킷 정책.
- **D.** Enabling AWS Backup to snapshot the bucket contents daily.
  - 한글: AWS Backup으로 버킷 콘텐츠를 매일 스냅샷하도록 활성화.
### ✅ Correct Answer / 정답
**B**

### 💡 Explanation / 해설

**English**:
S3 Block Public Access is the safety switch that overrides any bucket or object permission granting public access. Applied at the bucket or account level, it prevents both existing and future misconfigurations from exposing data — a critical baseline control for training data and model artifacts.

- A (IAM Access Analyzer) reports on external access findings but does not enforce the block. It is a useful complement to BPA, not a substitute.
- C is the opposite of the protective control — a bucket policy that grants read to `"*"` is exactly the misconfiguration BPA is designed to override.
- D (AWS Backup) creates snapshots for recovery. It has no bearing on who can access the live bucket.

**한글**:
S3 Block Public Access(BPA)는 퍼블릭 접근을 허용하는 버킷·객체 권한을 무효화하는 안전 스위치입니다. 버킷 또는 계정 수준에 적용하면 기존/미래의 오설정이 데이터를 노출하지 못하도록 막습니다. 학습 데이터·모델 아티팩트를 위한 필수 기본 통제입니다.

- A(IAM Access Analyzer)는 외부 접근 findings를 보고하지만 차단을 강제하지 않습니다. BPA를 보완하는 도구이지 대체재가 아닙니다.
- C는 오히려 반대 — `"*"` 주체에 읽기를 부여하는 버킷 정책은 BPA가 무효화하려는 오설정입니다.
- D(AWS Backup)는 복구를 위한 스냅샷을 생성합니다. 실시간 버킷 접근 통제와는 무관합니다.

### 🇺🇸 Question (English)
Which BEST describes a "model extraction" (or model theft) attack against a deployed ML API?

### 🇰🇷 문제 (한글)

### Options / 보기

  - 한글: 공격자가 많은 쿼리를 보내 모델의 동작을 재구성·근사함.
- **B.** An attacker compromises the deployment host and overwrites the model file on disk.
  - 한글: 공격자가 배포 호스트를 침해하여 디스크의 모델 파일을 덮어씀.
- **C.** An attacker exploits a container escape to read model weights from the container filesystem.
  - 한글: 공격자가 컨테이너 이스케이프를 악용하여 컨테이너 파일시스템에서 모델 가중치를 읽음.
- **D.** An attacker floods the endpoint with malformed inputs to cause out-of-memory crashes.
  - 한글: 공격자가 형식이 잘못된 입력으로 엔드포인트를 과부하시켜 OOM 크래시를 유발.

### ✅ Correct Answer / 정답
**A**

### 💡 Explanation / 해설

**English**:
Model extraction is a query-based attack: the attacker treats the API as a black box, sends a large number of carefully-chosen inputs, records the outputs, and trains a surrogate model that mimics the target. The stolen surrogate then enables IP misuse or feeds into more targeted adversarial attacks. Mitigations focus on limiting information flow through the API: rate limits, per-caller quotas, output rounding or confidence-band truncation, anomalous-query detection, and requiring authentication.

- B describes a host-compromise scenario. If the attacker already has write access to the model file, they have gone far beyond extraction and can perform arbitrary tampering — a distinct attack class handled by host hardening, IAM, and encrypted artifacts.
- C describes a container-escape scenario. Reading weights directly from disk bypasses the API entirely. Defense is at the runtime/isolation layer, not at the API surface where extraction attacks operate.
- D describes a denial-of-service pattern. It disrupts availability but does not steal the model.

**한글**:

A multi-tenant SaaS uses Bedrock. Which pattern MOST reliably prevents one tenant's data from leaking into another tenant's responses?

멀티테넌트 SaaS가 Bedrock을 사용합니다. 한 테넌트의 데이터가 다른 테넌트의 응답으로 유출되지 않도록 가장 신뢰성 있게 방지하는 패턴은 무엇입니까?

- **C.** Route every tenant's query to the largest model to guarantee answer quality.
  - 한글: 답변 품질을 보장하기 위해 모든 테넌트 쿼리를 가장 큰 모델로 라우팅.

- C addresses answer quality, not isolation. A larger model provides no isolation guarantee; it can still be given another tenant's context by an unscoped retrieval layer.

- D는 모든 테넌트를 단일 지식 베이스로 붕괴시켜, 그것에 대한 검색이 다른 테넌트의 문서를 호출자에게 반환할 수 있게 만듭니다. 정확히 회피해야 할 안티 패턴입니다.

## Q362

### 🇺🇸 Question (English)
Beyond cost control, which SECURITY reason motivates enforcing per-user rate limits on a public GenAI endpoint?

### 🇰🇷 문제 (한글)
비용 통제 외에, 공용 GenAI 엔드포인트에 사용자별 레이트 리밋을 강제하는 보안 상의 동기는 무엇입니까?

### Options / 보기
- **A.** Rate limits force attackers to buy faster hardware, which exposes their identity.
  - 한글: 레이트 리밋이 공격자에게 더 빠른 하드웨어 구매를 강제하여 정체를 노출시킴.
- **B.** Rate limits slow abuse patterns and make them easier to detect and respond to.
  - 한글: 레이트 리밋이 남용 패턴을 둔화시켜 탐지·대응을 쉽게 만듦.
- **C.** Rate limits automatically block all known malicious IP addresses at the network edge.
  - 한글: 레이트 리밋이 알려진 악성 IP를 네트워크 엣지에서 자동으로 차단.
- **D.** Rate limits encrypt the model weights so that any extraction becomes useless.
  - 한글: 레이트 리밋이 모델 가중치를 암호화하여 추출을 무용지물로 만듦.

### ✅ Correct Answer / 정답
**B**

### 💡 Explanation / 해설

**English**:
Many abuse patterns rely on volume: model extraction needs many queries to fit a surrogate, credential stuffing needs many login attempts, denial-of-wallet needs many expensive completions, and prompt-injection probing needs many payload variants. Per-caller rate limits cap the volume any single identity can generate, which extends the attacker's timeline enough for anomaly detection and manual response to catch up. Rate limits do not eliminate the attack, but they change its economics.

- A is a fabricated causal chain; rate limits do not force any hardware choice and hardware speed is unrelated to identity exposure.
- C confuses rate limiting with network reputation controls (WAF, AWS Shield, GuardDuty). Rate limiting operates on request volume per identity, not on IP reputation lists.
- D confuses control planes. Rate limiting operates at the API layer; it does not perform encryption of any kind, and encryption of weights would not follow from throttling.

**한글**:
많은 남용 패턴은 볼륨에 의존합니다: 모델 추출은 대체 모델을 학습하기 위해 많은 쿼리를 요구하고, 크리덴셜 스터핑은 많은 로그인 시도를 요구하며, 지갑 소진은 많은 값비싼 완성을 요구하고, 프롬프트 인젝션 탐색은 많은 페이로드 변형을 요구합니다. 호출자별 레이트 리밋은 단일 아이덴티티가 생성할 수 있는 볼륨을 상한 설정하여 공격자의 타임라인을 늘리고, 이상 탐지와 수동 대응이 따라잡을 수 있게 합니다. 레이트 리밋이 공격을 없애지는 않지만, 공격의 경제성을 바꿉니다.

## Q369

### 🇺🇸 Question (English)
Which is a REAL use of IAM policy conditions to strengthen security for a SageMaker or Bedrock invocation role?

### 🇰🇷 문제 (한글)
SageMaker 또는 Bedrock 호출 역할의 보안을 강화하기 위한 IAM 정책 조건의 실제 사용 예는 무엇입니까?

### Options / 보기
- **A.** Restrict by source VPC endpoint, principal, resource, region, tag, or MFA presence.
  - 한글: 소스 VPC 엔드포인트, 주체, 리소스, 리전, 태그, MFA 여부로 제한.
- **B.** Allow all actions from any caller when the request path matches a naming convention.
  - 한글: 요청 경로가 명명 규칙과 일치하면 어떤 호출자의 모든 액션을 허용.
- **C.** Remove every identity-based policy and rely on network reachability alone.
  - 한글: 모든 아이덴티티 기반 정책을 제거하고 네트워크 도달성에만 의존.
- **D.** Grant AdministratorAccess to all users to eliminate access-denied errors during operations.
  - 한글: 운영 중 액세스 거부 오류를 없애기 위해 모든 사용자에게 AdministratorAccess 부여.

### ✅ Correct Answer / 정답
**A**

### 💡 Explanation / 해설

**English**:
IAM policy conditions narrow the context in which an otherwise-allowed action succeeds. Common levers for AI workloads: `aws:SourceVpce` to require calls come through a specific VPC endpoint, `aws:PrincipalTag`/`aws:ResourceTag` to require matching tenant or environment tags, `aws:RequestedRegion` to lock a role to specific regions, `aws:MultiFactorAuthPresent` to require MFA for sensitive operations, and resource-level conditions on model ARNs or endpoint ARNs. Combined, they turn a broad `Action` into a tightly-scoped effective permission.

- B is a naming-based access decision, which conditions cannot securely enforce — attackers control path strings, and naming has no cryptographic guarantee of identity.
- C removes identity-based control entirely. Network reachability decides "can bytes arrive", not "is this caller authorized"; the two must both be enforced.
- D is the antithesis of hardening. Broad admin access maximizes blast radius on any credential leak and removes the ability to use conditions meaningfully.

**한글**:
IAM 정책 조건은 이미 허용된 액션이 성공하는 컨텍스트를 좁힙니다. AI 워크로드의 흔한 레버: 특정 VPC 엔드포인트로만 호출되도록 요구하는 `aws:SourceVpce`, 일치하는 테넌트·환경 태그를 요구하는 `aws:PrincipalTag`/`aws:ResourceTag`, 특정 리전에 역할을 잠그는 `aws:RequestedRegion`, 민감 오퍼레이션에 MFA를 요구하는 `aws:MultiFactorAuthPresent`, 모델 ARN이나 엔드포인트 ARN에 대한 리소스 수준 조건. 이들을 결합하면 넓은 `Action`이 촘촘하게 스코핑된 유효 권한이 됩니다.

- B는 이름 기반 접근 결정이며 조건으로 안전하게 강제할 수 없습니다 — 공격자가 경로 문자열을 통제하고, 명명에는 아이덴티티의 암호학적 보장이 없습니다.
- C는 아이덴티티 기반 통제를 완전히 제거합니다. 네트워크 도달성은 "바이트가 도달할 수 있는가"이지 "이 호출자가 인가되었는가"가 아닙니다. 둘 다 강제해야 합니다.
- D는 강화의 정반대입니다. 넓은 관리자 접근은 자격 증명 유출 시 폭발 반경을 극대화하고, 조건을 의미 있게 사용할 여지를 없앱니다.

## Q370

### 🇺🇸 Question (English)
A hub account owns a Bedrock deployment; spoke accounts (different teams) need to invoke it. Which pattern is MOST secure?

### 🇰🇷 문제 (한글)
허브 계정이 Bedrock 배포를 소유하고, 스포크 계정(다른 팀들)이 이를 호출해야 합니다. 가장 안전한 패턴은 무엇입니까?

### Options / 보기
- **A.** Export IAM user access keys from the hub and email them to each spoke account owner.
  - 한글: 허브에서 IAM 사용자 액세스 키를 내보내 각 스포크 계정 소유자에게 이메일로 전송.
- **B.** Cross-account IAM role assumption scoped to minimum required actions and resources.
  - 한글: 필요한 최소 액션과 리소스로 스코핑된 크로스 계정 IAM 역할 assume.
- **C.** Distribute the hub account's root credentials to each spoke team lead as a shared password.
  - 한글: 허브 계정의 루트 자격 증명을 각 스포크 팀 리드에게 공유 비밀번호로 배포.
- **D.** Turn IAM off in the hub account so any authenticated spoke request is accepted.
  - 한글: 허브 계정의 IAM을 꺼서 인증된 스포크 요청은 모두 수락되게 함.

### ✅ Correct Answer / 정답
**B**

### 💡 Explanation / 해설

**English**:
The standard AWS cross-account pattern is role assumption: the hub account defines an IAM role with a trust policy naming the specific spoke accounts (or specific spoke roles), and a permissions policy scoped to only the actions and resources those spokes need. Spokes call `sts:AssumeRole` and use short-lived temporary credentials. Add `sts:ExternalId` for third-party trust relationships to prevent confused-deputy attacks, PrivateLink between accounts if traffic must stay off the internet, and CloudTrail in every account so each side's activity is auditable.

- A distributes long-lived credentials by email — an unauditable, easily-leaked channel — and violates least privilege since one exported key cannot be individually scoped per spoke consumer.
- C shares root credentials, which cannot be scoped, cannot be safely rotated across many humans, and grant unrestricted access to the entire hub account. AWS explicitly recommends against operational use of root.
- D is not a real capability — IAM cannot be disabled — and even conceptually treating "authenticated" as "authorized" removes the very distinction that authorization exists to make.

**한글**:
표준 AWS 크로스 계정 패턴은 역할 assume입니다. 허브 계정은 특정 스포크 계정(또는 특정 스포크 역할)을 명명한 신뢰 정책과, 스포크가 필요한 액션·리소스로 스코핑된 권한 정책을 가진 IAM 역할을 정의합니다. 스포크는 `sts:AssumeRole`을 호출해 단기 임시 자격을 사용합니다. 서드파티 신뢰에는 confused-deputy 공격을 방지하는 `sts:ExternalId`를 추가하고, 트래픽이 인터넷을 벗어나지 않아야 한다면 계정 간 PrivateLink를, 각 계정에서 활동을 감사할 수 있도록 CloudTrail을 활성화합니다.

- A는 이메일로 장기 자격 증명을 배포합니다 — 감사할 수 없고 쉽게 유출되는 채널이며, 하나의 내보낸 키는 스포크 소비자별로 개별 스코핑할 수 없어 최소 권한을 위반합니다.
- C는 루트 자격 증명을 공유합니다. 루트는 스코핑할 수 없고, 여러 사람 사이에서 안전하게 순환할 수 없으며, 허브 계정 전체에 무제한 접근을 부여합니다. AWS는 루트의 운영 사용을 명시적으로 억제합니다.
- D는 실제 기능이 아닙니다 — IAM은 끌 수 없습니다 — 개념적으로도 "인증됨"을 "인가됨"과 동일시하면 인가가 존재하는 이유 자체가 사라집니다.

### Options / 보기

  - 한글: 스토리지 비용을 없애고 데이터 보존 문제를 단순화하기 위해 아무것도 로깅하지 않음.

## Q372

### 🇺🇸 Question (English)
"Defense in depth" for an AI/ML workload MEANS:

### 🇰🇷 문제 (한글)
AI/ML 워크로드에 대한 "심층 방어(Defense in depth)"는 무엇을 의미합니까?

### Options / 보기
- **A.** Rely on one strong authentication method and skip the other layers as redundant.
  - 한글: 하나의 강력한 인증 방식에만 의존하고 다른 계층은 중복이라 여겨 생략.
- **B.** Encrypt data at rest and consider all other controls unnecessary duplication.
  - 한글: 저장 중 데이터를 암호화하고 다른 모든 통제는 불필요한 중복으로 간주.
- **C.** Enforce IAM policies only and defer network and application controls indefinitely.
  - 한글: IAM 정책만 강제하고 네트워크·애플리케이션 통제는 무기한 연기.
- **D.** Layer independent controls across identity, network, data, application, and operations.
  - 한글: 아이덴티티·네트워크·데이터·애플리케이션·운영에 걸쳐 독립적 통제를 계층화.

### ✅ Correct Answer / 정답
**D**

### 💡 Explanation / 해설

**English**:
Defense in depth means no single control is a single point of failure. Identity controls (IAM, IAM Identity Center) decide who may act. Network controls (VPC endpoints, PrivateLink, security groups) decide where traffic may go. Data controls (KMS at rest, TLS in transit, S3 Block Public Access, Macie) protect the payload. Application controls (Guardrails, input validation, rate limits, output validation) constrain what the model does. Operational controls (CloudTrail, CloudWatch, Inspector, Config, incident runbooks) detect and respond when any of the above fails. When one layer is bypassed, the others still hold.

- A collapses defense into a single control and inherits its every weakness. Modern attacks routinely bypass any one authentication method (phishing, session theft, token replay).
- B is the same failure through a different layer. Encryption at rest is defeated the moment credentials to decrypt are compromised, and does nothing against authorization or content-policy failures.
- C treats a permission-layer control as the entire security program. Nothing in IAM prevents a legitimately-authorized caller from injecting a prompt, exfiltrating data, or being compromised at the application layer.

**한글**:
심층 방어는 어떤 단일 통제도 단일 실패 지점이 되지 않는다는 뜻입니다. 아이덴티티 통제(IAM·IAM Identity Center)는 누가 행위할 수 있는지를 결정합니다. 네트워크 통제(VPC 엔드포인트·PrivateLink·보안 그룹)는 트래픽이 어디로 갈 수 있는지를 결정합니다. 데이터 통제(KMS 저장 중·TLS 전송 중·S3 Block Public Access·Macie)는 페이로드를 보호합니다. 애플리케이션 통제(Guardrails·입력 검증·레이트 리밋·출력 검증)는 모델이 하는 일을 제약합니다. 운영 통제(CloudTrail·CloudWatch·Inspector·Config·인시던트 런북)는 위 계층 중 어느 것이라도 실패했을 때 탐지·대응합니다. 한 계층이 우회되어도 나머지가 여전히 버팁니다.

- A는 방어를 하나의 통제로 붕괴시켜 그 통제의 모든 약점을 그대로 상속합니다. 최신 공격은 단일 인증 방식(피싱·세션 탈취·토큰 재생)을 일상적으로 우회합니다.
- B는 다른 계층에서 같은 실패입니다. 저장 시 암호화는 복호 자격이 침해되는 순간 무너지고, 인가나 콘텐츠 정책 실패에는 아무 역할도 하지 않습니다.
- C는 권한 계층 통제를 전체 보안 프로그램으로 취급합니다. 정당하게 인가된 호출자가 프롬프트를 주입하거나 데이터를 유출하거나 애플리케이션 계층에서 침해되는 것을 IAM이 막지 못합니다.
