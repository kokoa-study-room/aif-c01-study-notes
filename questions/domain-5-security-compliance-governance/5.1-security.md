## Q345

### 🇺🇸 Question (English)
Which principle is FUNDAMENTAL when configuring IAM for AI workloads?

### 🇰🇷 문제 (한글)
AI 워크로드에 대해 IAM을 설정할 때 근본적인 원칙은 무엇입니까?

### Options / 보기
- **A.** Principle of least privilege — IAM roles and policies grant ONLY the permissions required for the specific job or function, no more.
  - 한글: 최소 권한 원칙 — IAM 역할·정책이 특정 작업이나 기능에 **필요한 권한만** 부여하고 그 이상은 부여하지 않음.
- **B.** Always grant AdministratorAccess to every role.
  - 한글: 모든 역할에 AdministratorAccess 부여.
- **C.** Never use IAM policies.

- **D.** Hard-code credentials.

### ✅ Correct Answer / 정답
**A**

### 💡 Explanation / 해설

**English**:
Least privilege is the foundational IAM principle. B, C, D are severe anti-patterns.

**한글**:
최소 권한이 IAM의 기본 원칙입니다. B·C·D는 심각한 안티 패턴입니다.

## Q369

### 🇺🇸 Question (English)
Which is a REAL use of IAM policy conditions to strengthen security for a SageMaker or Bedrock invocation role?

### 🇰🇷 문제 (한글)
SageMaker 또는 Bedrock 호출 역할의 보안을 강화하기 위한 IAM 정책 조건의 실제 사용 예는 무엇입니까?

### Options / 보기
- **A.** Restrict access to specific source VPC endpoints, specific IAM principals, specific model ARNs, specific regions, tag-based conditions (e.g., matching tenant tag), or require MFA — narrowing the principals + context that can invoke the resource.
  - 한글: 특정 소스 VPC 엔드포인트, 특정 IAM 주체, 특정 모델 ARN, 특정 리전, 태그 기반 조건(예: 테넌트 태그 매칭), MFA 요구 등으로 제한 — 리소스를 호출할 수 있는 주체와 컨텍스트를 좁힘.
- **B.** Allow all actions from anywhere at all times.
  - 한글: 언제 어디서나 모든 액션 허용.
- **C.** Remove all IAM policies.
  - 한글: 모든 IAM 정책 제거.
- **D.** Grant AdministratorAccess to every user.
  - 한글: 모든 사용자에게 AdministratorAccess 부여.

### ✅ Correct Answer / 정답
**A**

### 💡 Explanation / 해설

**English**:
IAM policy conditions narrow context (VPC, region, principal, tags, MFA). B, C, D destroy security.

**한글**:
IAM 정책 조건은 컨텍스트(VPC·리전·주체·태그·MFA)를 좁힙니다. B·C·D는 보안을 파괴합니다.

## Q370

### 🇺🇸 Question (English)
A "hub" account owns a Bedrock deployment; "spoke" accounts (different teams) need to invoke it. Which pattern is MOST secure?

### 🇰🇷 문제 (한글)
"허브" 계정이 Bedrock 배포를 소유하고, "스포크" 계정(다른 팀들)이 이를 호출해야 합니다. 가장 안전한 패턴은 무엇입니까?

### Options / 보기
- **A.** Share IAM user access keys via email.
  - 한글: 이메일로 IAM 사용자 액세스 키 공유.
- **B.** Cross-account IAM roles (spoke roles assume hub roles), scoped to the minimum required actions/resources, with `sts:ExternalId` where applicable, PrivateLink between accounts if needed, and CloudTrail logging in each account.
  - 한글: 크로스 계정 IAM 역할(스포크 역할이 허브 역할을 assume), 최소 필요 액션·리소스로 스코프, 적용 가능한 경우 `sts:ExternalId` 사용, 필요 시 계정 간 PrivateLink, 각 계정에서 CloudTrail 로깅.
- **C.** Share the root user password.
  - 한글: 루트 사용자 비밀번호 공유.
- **D.** Disable IAM in both accounts.
  - 한글: 두 계정 모두 IAM 비활성화.

### ✅ Correct Answer / 정답
**B**

### 💡 Explanation / 해설

**English**:
Cross-account IAM roles with scoping and logging is the standard pattern. A, C, D are severe anti-patterns.

**한글**:
스코프와 로깅이 있는 크로스 계정 IAM 역할이 표준 패턴입니다. A·C·D는 심각한 안티 패턴입니다.

## Q371

### 🇺🇸 Question (English)
Which logging pattern BEST supports security investigations for a Bedrock-backed application?

### 🇰🇷 문제 (한글)
Bedrock 기반 애플리케이션의 보안 조사를 가장 잘 지원하는 로깅 패턴은 무엇입니까?

### Options / 보기
- **A.** Log nothing.
  - 한글: 아무것도 로깅하지 않음.
- **B.** Log only successful requests.
  - 한글: 성공 요청만 로깅.
- **C.** Enable CloudTrail (API activity), CloudWatch Logs (application logs), S3 access/data-event logging for training/artifact buckets, invocation-level tracing (request IDs), and Guardrail block events — all with retention appropriate to compliance requirements.
  - 한글: CloudTrail(API 활동), CloudWatch Logs(애플리케이션 로그), 학습·아티팩트 버킷의 S3 접근·데이터 이벤트 로깅, 호출 단위 트레이싱(요청 ID), Guardrail 차단 이벤트 활성화 — 모두 컴플라이언스 요구사항에 맞는 보존.
- **D.** Log only on Fridays.
  - 한글: 금요일에만 로깅.

### ✅ Correct Answer / 정답
**C**

### 💡 Explanation / 해설

**English**:
Comprehensive multi-layer logging + retention supports incident response. A, B, D leave gaps.

**한글**:
포괄적인 다층 로깅 + 보존이 인시던트 대응을 지원합니다. A·B·D는 공백을 남깁니다.

## Q372

### 🇺🇸 Question (English)
"Defense in depth" for an AI/ML workload MEANS:

### 🇰🇷 문제 (한글)
AI/ML 워크로드에 대한 "심층 방어(Defense in depth)"는 무엇을 의미합니까?

### Options / 보기
- **A.** Relying on a single security control everywhere.
  - 한글: 모든 곳에서 하나의 보안 제어에 의존.
- **B.** Only encrypting data.
  - 한글: 데이터 암호화만.
- **C.** Only using IAM.
  - 한글: IAM만 사용.
- **D.** Layering multiple independent controls — identity (IAM/Identity Center), network (VPC endpoints, PrivateLink, security groups), data (KMS at rest, TLS in transit, S3 Block Public Access), application (Guardrails, input validation, rate limits), operations (CloudTrail, monitoring, Inspector, Config, incident runbooks) — so a single failure doesn't compromise the system.
  - 한글: 여러 독립적 제어를 계층화 — 아이덴티티(IAM/Identity Center), 네트워크(VPC 엔드포인트·PrivateLink·보안 그룹), 데이터(KMS 저장 중·TLS 전송 중·S3 Block Public Access), 애플리케이션(Guardrails·입력 검증·레이트 리밋), 운영(CloudTrail·모니터링·Inspector·Config·인시던트 런북) — 단일 실패가 시스템 전체를 위협하지 않도록.

### ✅ Correct Answer / 정답
**D**

### 💡 Explanation / 해설

**English**:
Defense in depth = multiple independent layers across identity, network, data, app, ops. A, B, C are single-layer.

**한글**:
심층 방어 = 아이덴티티·네트워크·데이터·앱·운영에 걸친 다중 독립 계층. A·B·C는 단일 계층입니다.
