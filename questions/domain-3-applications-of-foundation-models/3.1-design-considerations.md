## Q177

### 🇺🇸 Question (English)
A team must select a foundation model for a real-time chat feature with strict sub-second latency requirements. Which trade-off consideration is MOST relevant?

### 🇰🇷 문제 (한글)
초 미만 지연 요구가 엄격한 실시간 채팅 기능을 위해 파운데이션 모델을 선택해야 합니다. 가장 관련성 높은 트레이드오프 고려사항은 무엇입니까?

### Options / 보기
- **A.** Larger models often have higher quality but longer inference latency and higher per-token cost — for real-time UX, a smaller/faster model may be a better fit if accuracy is acceptable.
  - 한글: 큰 모델은 대개 품질이 높지만 추론 지연·토큰당 비용도 큽니다. 실시간 UX에서는 정확도가 허용 가능하다면 작고 빠른 모델이 더 적합할 수 있습니다.
- **B.** Always choose the largest available model regardless of latency.
  - 한글: 지연과 무관하게 항상 가장 큰 모델을 선택.
- **C.** Model size does not affect latency at all.
  - 한글: 모델 크기는 지연에 전혀 영향을 주지 않음.
- **D.** Only image models can meet sub-second latency.
  - 한글: 초 미만 지연은 이미지 모델만 달성 가능.

### ✅ Correct Answer / 정답
**A**

### 💡 Explanation / 해설

**English**:
Model size trades quality vs cost/latency. Real-time UX often prefers smaller models when quality is sufficient. B, C, D contradict basic engineering realities.

**한글**:
모델 크기는 품질 vs 비용/지연을 트레이드합니다. 실시간 UX는 품질이 충분하다면 작은 모델을 선호합니다. B·C·D는 기본 엔지니어링 사실에 반합니다.

### 🇺🇸 Question (English)
A user-facing GenAI feature has many repeated identical or near-identical prompts (e.g., FAQ-style questions). Which pattern MOST reduces cost and latency?

### 🇰🇷 문제 (한글)
사용자 대상 GenAI 기능에서 동일하거나 매우 유사한 프롬프트가 자주 반복됩니다(예: FAQ 스타일 질문). 비용과 지연을 가장 크게 줄이는 패턴은 무엇입니까?

### Options / 보기
- **A.** Increase model temperature to reduce repetition.
  - 한글: 반복을 줄이려 model temperature를 상향.
- **B.** Always call the largest model twice per request.
  - 한글: 요청마다 가장 큰 모델을 두 번씩 호출.
- **C.** Disable rate limits.
  - 한글: 레이트 리밋 비활성화.

**한글**:
응답 캐싱(정확 또는 시맨틱)은 반복 쿼리 비용/지연을 줄이는 표준 최적화입니다. A·B·C는 상황을 악화시킵니다.

When is "prompt chaining" (breaking a task into multiple LLM calls) BETTER than a single monolithic prompt?

### 🇰🇷 문제 (한글)
"프롬프트 체이닝"(작업을 여러 LLM 호출로 분할)이 단일 monolithic 프롬프트보다 나은 시점은 언제입니까?

- **A.** Always — chaining is universally better.
  - 한글: 항상 — 체이닝이 언제나 더 나음.
- **B.** Never — chaining is always worse.
  - 한글: 절대 — 체이닝은 항상 더 나쁨.
- **C.** When a complex task decomposes naturally into staged sub-tasks (e.g., extract → classify → summarize), each with a smaller, more reliable prompt and clearer evaluation.

**C**

### 💡 Explanation / 해설

**English**:
Chaining helps when sub-tasks are cleanly separable and each step improves reliability/evaluability. A and B are dogmatic. D is unrelated.

**한글**:
하위 작업이 깔끔히 분리되고 각 단계가 신뢰성/평가 가능성을 개선할 때 체이닝이 유리합니다. A·B는 교조적, D는 무관합니다.

## Q185

### 🇺🇸 Question (English)
When calling a Bedrock model that occasionally returns transient throttling errors, which retry strategy is BEST?

### 🇰🇷 문제 (한글)
가끔 일시적 쓰로틀링 오류를 반환하는 Bedrock 모델을 호출할 때 가장 좋은 재시도 전략은 무엇입니까?

### Options / 보기
- **A.** Exponential backoff with jitter, bounded retry count, and clear surfacing of persistent failures — retrying transient errors but not permanent ones.

- **C.** Never retry any error.
  - 한글: 어떤 오류도 재시도하지 않음.
- **D.** Delete the model on the first failure.
  - 한글: 첫 실패 시 모델 삭제.

### ✅ Correct Answer / 정답
**A**

**English**:

**한글**:
클라우드 표준 관행: 지연·지수 백오프 + jitter, 일시적 vs 영구 오류 구분. B는 재시도 폭주 유발, C는 회복 가능 실패를 놓침, D는 터무니없습니다.
