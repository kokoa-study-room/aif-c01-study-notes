## Q081

### 🇺🇸 Question (English)
Which pair of statements BEST distinguishes a generative model from a discriminative model?

### 🇰🇷 문제 (한글)
생성 모델과 판별 모델을 가장 잘 구분하는 진술의 짝은 무엇입니까?

### Options / 보기
- **A.** Generative models learn P(x) or P(x,y) — modeling how data is generated so they can produce new samples; discriminative models learn P(y|x) — mapping inputs to labels for classification/regression.
  - 한글: 생성 모델은 P(x) 또는 P(x,y)를 학습해 데이터 생성 방식을 모델링하고 새 샘플을 만들 수 있으며, 판별 모델은 P(y|x)를 학습해 입력을 라벨에 매핑(분류/회귀)합니다.
- **B.** Generative models require no data; discriminative models require infinite data.
  - 한글: 생성 모델은 데이터가 필요 없고, 판별 모델은 무한한 데이터를 요구합니다.
- **C.** Generative and discriminative models are two names for the same technique.
  - 한글: 생성 모델과 판별 모델은 같은 기법을 부르는 두 이름입니다.
- **D.** Generative models only work on images; discriminative models only work on text.
  - 한글: 생성 모델은 이미지에만, 판별 모델은 텍스트에만 동작합니다.

### ✅ Correct Answer / 정답
**A**

### 💡 Explanation / 해설

**English**:
Generative models capture the data distribution so they can produce new samples (text, images, audio). Discriminative models learn decision boundaries mapping inputs to labels. B, C, and D invent false constraints.

**한글**:
생성 모델은 데이터 분포를 학습하여 텍스트·이미지·오디오 등 새 샘플을 생성합니다. 판별 모델은 입력을 라벨에 매핑하는 결정 경계를 학습합니다. B·C·D는 사실이 아닙니다.

"파운데이션 모델"을 전통적인 단일 작업 ML 모델과 가장 크게 구분 짓는 특성은 무엇입니까?

### Options / 보기

  - 한글: 파운데이션 모델은 광범위한 대규모 데이터로 한 번 학습된 뒤, 프롬프팅·파인튜닝·RAG로 다양한 하위 작업에 적응됩니다.

## Q083

### 🇺🇸 Question (English)
A developer asks an LLM: "Translate 'good morning' to Korean." with no examples provided in the prompt. What kind of prompt is this?

### 🇰🇷 문제 (한글)
한 개발자가 프롬프트에 예시를 전혀 제공하지 않고 LLM에 "'good morning'을 한국어로 번역해줘"라고 요청합니다. 이는 어떤 종류의 프롬프트입니까?

### Options / 보기
- **A.** Few-shot prompt
  - 한글: 퓨샷 프롬프트
- **B.** One-shot prompt
  - 한글: 원샷 프롬프트
- **C.** Zero-shot prompt
  - 한글: 제로샷 프롬프트
- **D.** Chain-of-thought prompt

### ✅ Correct Answer / 정답
**C**

### 💡 Explanation / 해설

**English**:
Zero-shot = no examples, just the request. A provides several examples. B provides exactly one example. D asks the model to reason step by step, orthogonal to shot-count.

**한글**:
제로샷은 예시 없이 요청만 주는 것입니다. A는 여러 예시, B는 정확히 하나의 예시, D는 단계별 추론을 유도하며 예시 개수와 별개 개념입니다.

## Q085

### 🇺🇸 Question (English)
Which situation BEST benefits from few-shot prompting rather than zero-shot?

### 🇰🇷 문제 (한글)
제로샷 대신 퓨샷 프롬프팅이 가장 도움이 되는 상황은 언제입니까?

### Options / 보기
- **A.** When the task follows a specific format or style the model may not infer from the instruction alone (e.g., custom classification labels, unusual output structure), and providing several concrete examples helps the model match the pattern.
  - 한글: 작업이 지시만으로 모델이 추론하기 어려운 특정 형식·스타일을 따를 때 (예: 커스텀 분류 라벨, 특이한 출력 구조), 여러 구체 예시가 패턴 매칭을 도와줄 때.
- **B.** When compute cost per token is not a concern and infinite examples are always better.
  - 한글: 토큰당 컴퓨트 비용이 문제가 되지 않고 예시가 많을수록 항상 좋을 때.
- **C.** When the task is a simple arithmetic problem the model already solves perfectly.
  - 한글: 모델이 이미 완벽히 푸는 단순 산술 문제일 때.
- **D.** When you want the model to ignore all provided examples.
  - 한글: 모델이 제공된 모든 예시를 무시하기를 원할 때.

### ✅ Correct Answer / 정답
**A**

### 💡 Explanation / 해설

**English**:
Few-shot shines when the model needs concrete examples of the desired output structure or unusual label set. B is wrong — more tokens cost more and past a point add noise. C would waste tokens. D is nonsensical.

**한글**:
퓨샷은 원하는 출력 구조나 특이한 라벨 집합의 구체 예시를 모델이 참고해야 할 때 빛납니다. B는 예시가 많을수록 비용·잡음이 증가하므로 오답, C는 이미 잘 푸는 작업이라 낭비, D는 무의미합니다.

## Q088

### 🇺🇸 Question (English)
Which BEST defines "in-context learning" in LLMs?

### 🇰🇷 문제 (한글)
LLM에서 "인컨텍스트 러닝(in-context learning)"을 가장 잘 정의한 것은 무엇입니까?

### Options / 보기
- **A.** Retraining model weights on new data at every request.
  - 한글: 매 요청마다 새 데이터로 모델 가중치를 재학습.
- **B.** Storing user chats in a database for analytics.
  - 한글: 사용자 대화를 분석용으로 데이터베이스에 저장.
- **C.** Manually rewriting the model's Python source code.
  - 한글: 모델의 Python 소스 코드를 수동으로 재작성.
- **D.** Adjusting the model's behavior on a task by adding examples and context INTO THE PROMPT, without changing weights.
  - 한글: 가중치를 바꾸지 않고, 프롬프트에 예시와 문맥을 추가하여 특정 작업에 대한 모델 동작을 조정.

**English**:

**한글**:
인컨텍스트 러닝은 프롬프트에 담긴 예시·지시만으로(가중치 변경 없이) 작업을 수행하도록 모델 동작을 조정하는 것입니다. A는 파인튜닝과 혼동, B·C는 무관합니다.

## Q090

### 🇺🇸 Question (English)
Which statement BEST describes the practical difference between `temperature` and `Top-P`?

### 🇰🇷 문제 (한글)
`temperature`와 `Top-P`의 실질적 차이를 가장 잘 설명한 것은 무엇입니까?

### Options / 보기
- **A.** They are the same knob, just with different names.
  - 한글: 이름만 다를 뿐 동일한 조절 손잡이입니다.
- **B.** Temperature scales probabilities across ALL candidate tokens (controls overall randomness), while Top-P restricts sampling to the smallest set of tokens whose cumulative probability meets P (controls the token pool adaptively).
  - 한글: Temperature는 모든 후보 토큰의 확률을 스케일링(전체 무작위성 제어)하고, Top-P는 누적 확률이 P가 되는 가장 작은 토큰 집합으로 샘플링을 제한(토큰 풀을 적응적으로 제어)합니다.
- **C.** Temperature affects only image models; Top-P affects only text models.
  - 한글: Temperature는 이미지 모델에만, Top-P는 텍스트 모델에만 영향을 줍니다.
- **D.** Both must always be set to 0 in production.
  - 한글: 프로덕션에서는 둘 다 항상 0으로 설정해야 합니다.

### ✅ Correct Answer / 정답
**B**

### 💡 Explanation / 해설

**English**:
Temperature and Top-P are complementary randomness controls. Temperature reshapes the whole distribution; Top-P truncates the tail. A is false. C invents a modality constraint. D would kill any diversity.

**한글**:
Temperature와 Top-P는 상호 보완적인 무작위성 제어 손잡이입니다. Temperature는 전체 분포를 재조정하고, Top-P는 꼬리를 잘라냅니다. A는 사실 아님, C는 지어낸 모달리티 제약, D는 다양성을 완전히 없앱니다.
